# win-auto-native Documentation

## Overview

`win-auto-native` is a Rust library providing Windows automation capabilities through
Node.js N-API bindings (napi-rs). It wraps Windows UI Automation (UIA), Win32 APIs,
COM, and OLE into async functions consumed by the TypeScript `@win-auto/core` package.

## Architecture

The library compiles to a `cdylib` (.node binary). Each Rust source file in `src/`
corresponds to a capability module, exported via `#[napi]` macros.

### Key Dependencies

- **napi** (v2) / **napi-derive** (v2): N-API bindings
- **tokio** (v1): Async runtime for non-blocking exports
- **windows** (v0.62): Safe bindings to Win32 and COM APIs
- **image** (v0.25): PNG encoding/decoding for screenshots and template matching
- **rayon** (v1): Parallel template matching (quadrant NCC search)
- **tracing**: Structured logging for debugging

## Module Reference

### `config.rs`
Exports `setAppConfig(executable, classNames)` — configures class-name filters for
specific executables.

### `dialogs.rs`
Dialog detection and interaction:
- `findDialogs(processId)` — enumerate modal dialogs for a PID
- `getDialogControls(windowHandle)` — list dialog buttons and inputs
- `clickDialogButton(windowHandle, buttonText)` — click button by text
- `setDialogFilePath(windowHandle, path)` — type path in file-open dialogs

### `discovery.rs`
UIA-based element and window discovery:
- `find_element_uia_by_conditions` — fast path + slow path with LegacyIAccessible fallback
- `discover_windows_for_pid` — all top-level windows for a process
- `collect_all_top_level_windows` — enumerate all top-level HWNDs
- `inspect_window_tree`, `inspect_hwnd_tree` — recursive tree walking

### `error.rs`
`AutomationError` enum (11 variants, `thiserror`-based):
- `InvalidHandle`, `ElementNotFound`, `WindowNotFound`, `ComInitFailed`
- `PermissionDenied`, `PatternNotSupported`, `ProcessLaunchFailed`
- `ScreenshotFailed`, `Timeout`, `DialogFailed`, `Generic`
Each variant maps to the appropriate napi-rs `Status` code.

### `event_watcher.rs`
WinEvent hooks (P9.2):
- `startWinEventWatcher(callback)` — spawns a background thread with `SetWinEventHook`
  monitoring `EVENT_OBJECT_CREATE`, `DESTROY`, `SHOW`, `HIDE`, `FOCUS`, `VALUECHANGE`,
  `EVENT_SYSTEM_MENUSTART`, `MENUEND`
- `stopWinEventWatcher()` — un hooks and joins the thread
- Uses `ThreadsafeFunction` to invoke the JS callback from the background thread

### `highlight.rs`
Visual element highlighting:
- `highlightElement(handle, color, durationMs)` — draws a colored border overlay

### `interaction.rs`
User-input simulation:
- `clickElement`, `rightClickElement`, `doubleClickElement`, `hoverElement`
- `typeText` (via `WM_SETTEXT` + `EM_SETSEL`) / `sendKeys` (via `SendInput`)
- `mouseMove`, `scrollElement`, `pressKey`, `keyDown`, `keyUp`
- `selectText`, `getSelection`, `replaceSelectedText`
- `dragDrop` (delegates to `ole_drag.rs`)
- `pressKeyCodes`, `clickSequence`, `clickElementByName`
- `getValue`, `setValue`, `getToggleState`, `getElementAttribute`

### `ole_drag.rs`
OLE-aware drag-drop (P9.3):
- `mouse_simulation_drag_drop(from, to, try_ole)` — 30-step smooth mouse animation
- When `try_ole=true`, initializes OLE on the worker thread so the OS dispatches
  proper OLE drag-drop events during mouse simulation

### `process_control.rs`
Process lifecycle (P9.4):
- `launch(path, classNames?)` — spawn via `CreateProcessW` + job object, returns PID
- `launchProcess(path, options?)` — full-featured launch with args/cwd/env
- `closeApp(pid)` — graceful WM_CLOSE → force TerminateProcess
- `closeWindow(handle)` — UIA WindowPattern.Close() → PostMessage WM_CLOSE
- `isProcessRunning`, `killProcess`, `findProcessesByName`
- `waitForProcessExit`, `getProcessImageName`, `enumerateWindows`

### `screenshot.rs`
Screen capture and template matching (P9.5):
- `captureScreenshot(handle)` — GDI `BitBlt` → BGRA → PNG bytes
- `captureScreenshotToFile(handle, path)` — save PNG to disk
- `findImage(handle, templatePng)` — parallel NCC quadrant search via `rayon`
  with coarse (step=2) → refine pass

### `trace.rs`
Tracing initialization:
- `init_tracing()` — one-time subscriber setup for `tracing` events

### `utils.rs`
Shared helpers:
- `ComScope` — refcount-based thread-local COM initialization (`CoInitializeEx`/`CoUninitialize`)
- `parse_hwnd`, `hwnd_to_string` — handle conversion
- `get_class_name`, `get_window_title`, `is_visible`, `window_pid`
- `process_image_for_pid`, `get_dpi_for_window`

## NAPI Exports

All public functions are auto-generated by napi-rs and exposed as async/sync JS functions.
See the TypeScript type definitions in `index.d.ts` for the full export list.

## Safety Considerations

- **Unsafe COM calls**: Every UIA/COM call is wrapped in `unsafe { }` blocks with SAFETY comments
- **Handle lifecycle**: Process/thread/window handles are explicitly managed (`CloseHandle`, `ReleaseDC`)
- **Thread safety**: COM is per-thread; `ComScope` maintains a thread-local refcount
- **OLE initialization**: Called per-worker-thread to avoid COM apartment conflicts
- **No Send/Sync on raw HANDLEs**: Job-object handles stored as `isize` to satisfy `Send` bounds

## Requirements

- Windows 10/11 (UIA + WinEvent APIs)
- Rust toolchain (for building from source)
